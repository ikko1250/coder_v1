# 2026-03-08 Main Rs Cleanliness Review

- 対象: `src/main.rs`
- 目的: コード設計のクリーンさ（可読性、責務分離、まとまり）の観点でレビュー
- 現状把握:
  - 単一ファイル内に CSV 読み込み、タグ解析、フィルタ定義、アプリ状態、UI 描画、日本語フォント設定、エントリポイントが同居している
  - `App` がデータ保持、フィルタ適用、選択移動、スクロール要求、キャッシュ、UI 描画をまとめて持つ構成
  - 一覧と詳細の 2 ペイン UI で、左側にフィルタとレコード一覧、右側に選択段落の詳細を表示する
- 主なレビュー観点:
  - 責務が分離されているか
  - 列定義やフィルタ定義が一箇所に集約されているか
  - 状態遷移が読み取りやすいか
  - UI 描画コードがロジックと過度に混在していないか
- 主要所見:
  - `main.rs` 1 ファイルに責務が集まりすぎており、追跡コストが高い
  - `FilterColumn` 追加時の変更箇所が分散しており、設計上の一貫性が弱い
  - 選択状態とスクロール状態の更新が複数メソッドに散っていて、状態遷移を追いづらい
  - `draw_tree` が描画、データ整形、クリック処理、選択確定、スクロール消費を同時に扱っている
- 出力方針:
  - レビュー回答では既存挙動の要約を先に置く
  - その後、重要度順に指摘と改善方向を提示する
  - コード変更は行わない
- 追加依頼対応:
  - 修正実装に入る前段として、具体的な修正の骨子を作成する
  - 骨子では「既存挙動」「分割単位」「変更ステップ」「テスト観点」「コミット単位」を明示する
  - 着手順は、挙動を変えにくい定義集約から始め、次に状態管理の整理、最後にファイル分割へ進める方針とする
- セカンドオピニオン反映:
  - 第1段階では `MatchedCategories` を単一値列と同列に扱わず、多値列として型で分ける前提を置く
  - 一覧列定義では幅を `f32` で持たず、`egui_extras::Column` を返せる形で保持する方針を採る
  - フィルタ候補ソートでは数値列の辞書順問題をこの段階で解消対象に含める
  - 第2段階以降の 1-frame lag と境界チェックは別論点として明示し、第1段階では手を入れない
- 第1段階の承認用実装案:
  - 既存挙動は維持し、変更対象を `FilterColumn` / フィルタ候補生成 / 一覧列定義に限定する
  - `FilterColumnSpec` を導入し、ラベル、候補抽出、レコード照合用値取得、ソート種別を集約する
  - 多値列は `SingleValue` / `MultiValue` の区別を持つ補助 enum で扱い、`MatchedCategories` の特例分岐を散らさない
  - `TreeColumnSpec` を導入し、ヘッダ、`Column` 生成、セル文字列生成を 1 定義にまとめる
  - `draw_tree` のヘッダ・列幅・値配列は `TreeColumnSpec` 配列から構築するように置換する
  - `build_filter_options` と `record_matches_filters` は spec 経由で値取得するよう変更する
  - 数値列のフィルタ候補順は、数値化できるものを数値順、それ以外を文字列順で並べる
- 第1段階の実装結果:
  - `FilterValueKind`, `FilterSortKind`, `FilterColumnSpec`, `TreeColumnSpec` を `src/main.rs` に追加
  - `FilterColumn::label` は spec 参照へ変更し、列ラベル定義の重複を除去
  - `record_matches_filters` は `column.spec().matches(...)` 経由へ置換し、`MatchedCategories` の特例分岐を整理
  - `build_filter_options` は spec 配列を走査し、候補抽出とソートを spec 側へ集約
  - `sentence_count`, `match_group_count`, `annotated_token_count` は数値ソート対象として扱うよう変更
  - `TREE_HEADERS` を廃止し、一覧ヘッダ・列幅・セル値を `TREE_COLUMN_SPECS` に集約
  - `draw_tree` は `TREE_COLUMN_SPECS` を用いて列構築・ヘッダ描画・セル値描画を行う構成へ置換
  - 単一値列は trim 済み文字列、多値列は `category_values` を返す形で整理
- 検証状況:
  - ローカル環境では `cargo` が Linux 側 PATH に存在せず、Windows 側 `cargo.exe` 実行は sandbox 制約で拒否された
  - その後、`cargo` による確認はユーザー側で実施する方針へ切替
  - こちらでは静的な差分確認まで実施し、コンパイル確認は未実施
