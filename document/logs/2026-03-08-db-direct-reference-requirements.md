# 2026-03-08 DB Direct Reference Requirements

- 対象: CSV ビューアへの DB 直接参照機能追加
- 依頼内容:
  - CSV ベース表示を補完する機能として、元のデータベースファイルを直接参照し、テキスト確認を可能にする要件定義への意見整理
- 既存挙動:
  - 現在のアプリは CSV を読み込み、抽出済み段落を一覧表示し、選択レコードの詳細とハイライト付き本文を表示する
  - 詳細表示の本文は CSV 内の `paragraph_text` / `paragraph_text_tagged` に依存しており、DB の原文や前後コンテキストは参照しない
  - そのため、CSV 生成時の加工結果が正しいか、抽出元の前後文脈がどうなっているかは、このアプリ単体では確認できない
- 要件定義上の主要論点:
  - 目的は「CSV の検証」と「前後コンテキスト確認」の 2 系統であり、UI と検索キーを分けて考えるべき
  - 「どのキーで DB の原文へ辿るか」が最重要で、`paragraph_id` 単独で足りるのか、`document_id` や連番が要るのかを先に確定する必要がある
  - コンテキスト表示は「前後何件/何文/何段落」を見せるのかを定義しないと実装判断がぶれる
  - CSV と DB の不一致時にどう見せるかを要件に含めるべき
- 提案方針:
  - 要件を「参照対象」「照合キー」「表示単位」「不一致時の扱い」「性能要件」「UI 配置」に分解して定義する
  - 初期スコープは読み取り専用に限定し、編集・再抽出・再分析までは含めない
- DB 確認結果:
  - `asset/ordinance_analysis3.db` の `analysis_paragraphs` テーブル存在を確認
  - カラムは `paragraph_id`, `run_id`, `document_id`, `paragraph_no`, `paragraph_text`, `created_at`
  - `paragraph_id` は主キー、`(document_id, paragraph_no)` は UNIQUE
  - そのため、対象段落の特定は `paragraph_id` で可能だが、前後段落取得は `document_id` と `paragraph_no` を使う設計が自然
- 設計上の意見:
  - 右ペインに DB 参照ボタンを置き、別ウィンドウで原文と前後 2 段落を表示する案は、現行 UI を崩さず導線を追加できるため妥当
  - 5 段落固定表示は MVP として適切
  - ただし「中心段落の特定キー」と「前後段落の取得キー」は分けて要件化すべき
  - 具体的には、起点は `paragraph_id`、周辺取得は `document_id + paragraph_no` 範囲検索とする
- 文書化対応:
  - `document/db-direct-reference-design.md` を新規作成する
  - 内容は、背景、機能要件、UI仕様、データ取得仕様、例外仕様、実装手順を含む
  - 初期スコープは段落単位に限定し、文単位参照は将来拡張として明記する
- セカンドオピニオン反映メモ:
  - `rusqlite` 採用案は妥当だが、実装時には使用方法調査を行ったうえで導入する
  - 初手は UI 実装ではなく、DB 読み取り層と状態モデルの確定を優先する
  - DB パスは固定文字列埋め込みより、アプリ基準で解決する方針を検討対象に含める
  - `paragraph_no` の下限ガードと、末尾移動時に中心段落が取得できないケースの扱いを要件へ反映済みとして実装時にも維持する
  - DB 参照ウィンドウは、メイン選択とは独立して開いた時点の対象を保持する方針が自然
- Step 1 実装:
  - `rusqlite` の公式ドキュメントで `Connection::open_with_flags` と `OpenFlags::SQLITE_OPEN_READ_ONLY` の使用方法を確認
  - `Cargo.toml` に `rusqlite` を追加
  - `src/model.rs` に `DbParagraph`, `DbParagraphContext` を追加
  - `src/db.rs` を新規作成し、読み取り専用接続、中心段落取得、前後段落取得、初期 DB パス解決を実装
  - 前後段落取得では `paragraph_no` 下限を `1` に clamp するよう実装
  - `src/main.rs` に `mod db;` を追加
- Step 1 現時点の方針:
  - 既存 UI はまだ変更しない
  - DB クエリ層は次段階の UI 実装から利用する前提で追加
  - コンパイル確認は今回もユーザー側 `cargo` 実行前提
- Step 2 実装:
  - `src/model.rs` に `DbViewerState` を追加
  - `DbViewerState` には、ウィンドウ開閉状態、DB パス、参照元 `paragraph_id`、CSV 段落テキスト、取得済みコンテキスト、エラーを保持する構成を採用
  - `src/app.rs` の `App` に `db_viewer_state` フィールドを追加
  - `App::new()` では `resolve_default_db_path()` を使って既定 DB パスを初期化
  - `load_csv()` では CSV 切替時に DB ビューア状態をリセットするよう変更
  - 次段階のボタン実装に備え、`selected_paragraph_id_for_db()` と `prepare_db_viewer_state()` を追加
- Step 2 現時点の方針:
  - UI はまだ変更しない
  - DB ビューア状態はメイン選択から独立した状態として保持する
  - 次段階で右ペインの `DB参照` ボタンから `prepare_db_viewer_state()` を呼ぶ前提
- Step 3 実装:
  - `src/app.rs` の右ペイン詳細表示に `DB参照` ボタンを追加
  - レコード選択時のみ有効、未選択時は無効ボタンとして表示
  - ボタン押下時は `prepare_db_viewer_state()` を呼び、失敗時は既存 `error_message` ダイアログへ流す構成にした
- Step 3 現時点の方針:
  - この段階では別ウィンドウはまだ描画しない
  - あくまでメイン画面から DB 参照状態を起動できる導線だけを追加
- Step 4 実装:
  - `src/app.rs` に `draw_db_viewer_window()` を追加
  - `update()` から DB 参照ウィンドウ描画を呼ぶよう変更
  - `DB参照` ボタン押下時は `prepare_db_viewer_state()` の後に初回コンテキスト取得を行うよう変更
  - ウィンドウには DB パス、中心段落メタ情報、CSV テキスト、DB 中心段落、前後コンテキストを表示する構成を追加
  - CSV と DB 中心段落の単純一致/不一致表示を追加
  - DB 取得失敗時はウィンドウ内にエラーを表示する構成にした
- Step 4 現時点の方針:
  - 前後移動ボタンはまだ追加しない
  - Step 4 は「開く」「初回取得」「比較表示」「前後 5 段落表示」に限定する
- Step 5 実装:
  - `src/db.rs` に `fetch_paragraph_context_by_location()` を追加し、`document_id + paragraph_no` から中心段落を含む前後コンテキストを取得できるようにした
  - `src/app.rs` に DB ビューア用の前後候補算出メソッドを追加
  - DB ウィンドウに `◀ 前へ` / `次へ ▶` ボタンと中心段落番号表示を追加
  - ボタン押下時は現在の `document_id + paragraph_no` を基準に再取得し、再描画する構成にした
  - 移動先取得に失敗した場合は古いコンテキストを破棄し、エラーメッセージを表示する
- Step 5 現時点の方針:
  - 前後移動は現在読み込んだコンテキスト内の隣接段落番号から決定する
  - DB 側段落番号に大きな欠番があるケースは、必要なら次段階で追加調整する
