# UI とアプリケーションコア分離・Tauri 移行ロードマップ 設計案

## 1. 文書の位置づけ

本書は、条例分析ビューア（現状: Rust + egui + eframe）について、**短期的な保守性向上**と**長期的な Tauri 移行の下地**を両立するための設計方針と段階計画をまとめたものである。実装タスクの詳細分解は別紙（タスク一覧・ログ）で行う。

## 2. 背景と目的

### 2.1 既存の状況

- `src/app.rs` に **アプリケーション状態・ドメイン処理・egui 描画**が大規模に同居しており、変更の影響範囲が読みにくい。
- 一部は既にビュー分離されている（`filter_panel_view`、`db_viewer_view`、`condition_editor_view` 等）が、**単一ファイルの肥大化**は継続している。
- `main.rs` は eframe の起動とフォント設定に特化している。

### 2.2 目的の整理

| 区分 | 目的 | 説明 |
|------|------|------|
| 短期 | **保守性の向上** | 責務の境界を明確にし、変更・レビュー・局所テストをしやすくする。 |
| 長期 | **Tauri への実移行** | WebView フロントへ UI を作り直す前提のため、**Rust 側に「UI に依存しないコア」を残す**。移行自体は準備が整うまで着手しない。 |

### 2.3 非目的（明示的にやらない／後回しでよいこと）

- **現行 egui UI と Tauri 版の機能を 1:1 で揃えること**は求めない。無駄機能の整理・UI の再設計を前提とする。
- **macOS 等のクロスプラットフォーム対応**は当面の設計要件に含めない（Windows のみ）。
- **起動時間・メモリの厳密な最適化**は優先度を下げる（開発用・個人環境想定。ただし応答性は別項）。

## 3. ステークホルダー要件（合意事項）

以下は要件定義として合意した内容である。

1. **長期目標**: Tauri への実移行。UI は再実装になる前提で、十分な準備のうえで着手する。
2. **短期目標**: 保守性向上（ロジックと描画の分離、モジュール境界の明確化）。
3. **機能パリティ**: 完全な一致は不要。再設計・削減を許容する。
4. **プラットフォーム**: 当面 **Windows のみ**を対象とする。
5. **非機能**:
   - 起動時間・総メモリは厳密な数値目標は設けない（**16GB 程度の環境で実用できる**こと）。
   - **応答速度を重視**し、UI 設計は **60 FPS を維持できる**ことを意識する。
6. **非同期・ジョブ**: 利用者は専門外のため、設計書側で**わかりやすいモデル**を提示し、実装はこの設計に沿う。
7. **Tauri 時の API 契約（コマンド／イベントの形）**: 実装側の提案に委ねる（本書で初期案を示す）。
8. **テスト**: いきなり全面カバレッジは狙わず、**少しずつ**足していく。
9. **移行**: **段階的**に進める。

## 4. 設計原則

### 4.1 レイヤ構造（目標像）

```
┌─────────────────────────────────────────────────────────┐
│  UI アダプタ層（交換可能）                                │
│  - 現状: egui / eframe（即時モード、毎フレーム描画）       │
│  - 将来: Tauri（フロント） + IPC / invoke                 │
└───────────────────────┬─────────────────────────────────┘
                        │ イベント入力 / 表示用状態の受け渡し
┌───────────────────────▼─────────────────────────────────┐
│  アプリケーションコア（UI 非依存を目指す）                 │
│  - ドメイン状態（レコード、フィルタ、選択、DB 参照状態等）   │
│  - 純粋に近い更新（入力イベント → 次状態・副作用の指示）    │
│  - 既存モジュールの再利用（csv_loader, db, filter 等）    │
└───────────────────────┬─────────────────────────────────┘
                        │ ファイル I/O、プロセス起動、DB 等
┌───────────────────────▼─────────────────────────────────┐
│  インフラ・副作用実行層（ホストが担当）                     │
│  - ファイル選択結果の反映、Python 分析プロセス、SQLite 等   │
└─────────────────────────────────────────────────────────┘
```

**原則**: `egui::Context`・`Color32`・`ViewportCommand` など **フレームワーク型はコアに持ち込まない**。コアは「何をしたいか」「状態はどう変わるか」を表現し、ホストが「描画・OS 対話」を行う。

### 4.2 長期（Tauri）と短期（egui）の関係

- 短期は **egui を継続**しつつ、**コアを太らせ、アダプタを薄くする**方向でよい。
- 長期 Tauri 移行時は **コア＋副作用インタフェースをそのままバックエンドに載せ**、フロントは新規実装。IPC の形は **serde 可能な構造体**に寄せておくと移行が楽になる。

### 4.3 機能パリティを求めないことの設計上の意味

- コアに切り出すのは「**残す価値がある業務ロジック**」に限定してよい。
- UI 専用の状態（スクロールアライメントの詳細など）は、**当面 egui アダプタに閉じてもよい**（無理にコアへ上げない）。ただしコアと重複する状態は **単一の情報源**に寄せる。

### 4.4 単一情報源と「分離で増えるバグ」の前提

コアと UI を分けると、**状態の二重化・更新順・イベントの遅延**により、従来単一 `App` 内で起きにくかった不整合が表に出やすくなる。§11 で列挙する脅威に対し、**コア側で不変条件（Invariant）をコードとテストで守る**ことを P1〜P2 の必須要件とする。

## 5. 非同期・ジョブモデル（わかりやすい説明）

### 5.1 何が問題か

分析処理（Python 子プロセスなど）は **すぐ終わらない**。一方 UI は **60 FPS で応答**させたい。そのため、**重い処理を UI スレッドで待たない**必要がある。

### 5.2 採用する考え方（概念モデル）

1. **バックグラウンド作業**  
   「分析を走らせる」など **時間のかかる処理**は、別スレッドまたはプロセスで実行する（現状の `mpsc` 受信に近い）。

2. **コアは「状態」と「届いた結果」だけを扱う**  
   コアは「**いまジョブは実行中か／完了したか／エラーか**」を表す。  
   実際の待受・チャネル読み取りは **ホスト（egui の `update` や Tauri のイベントループ）**が行い、**結果だけをコアに渡す**形に寄せる。

3. **進捗・通知**  
   ホストは「コアに結果を渡す」と同時に「**再描画が必要**」を知る（egui なら `request_repaint`、Tauri ならフロントへ `emit`）。

この 3 点を満たせば、「非同期の専門知識がなくても」**設計としては一貫**する。実装の詳細（チャネル名、スレッド数など）はコード側の責務とする。

### 5.3 ジョブ ID による結果の検証（必須）

バックグラウンド完了イベントが **遅延して届く**と、ユーザーがフィルタ変更・CSV 再読込・新規ジョブ開始の**後に古い結果が適用される**（レース）可能性がある。これは分離後に顕在化しやすいため、次を**設計要件**とする。

1. **すべての分析／エクスポートジョブに一意な `job_id`（または世代カウンタ）を付与**する。
2. コアは **「現在の有効ジョブ ID」**（または「期待する完了世代」）を保持する。
3. 完了・失敗・進捗イベントを受信したとき、**`job_id` が現在の有効なものと一致しない場合は破棄**し、状態を上書きしない。
4. 新しいジョブ開始・データソース切替時は、**世代を進める**（古いチャネルからの遅延メッセージを無効化できるようにする）。

§6 のイベント例にも `job_id` を必須フィールドとして載せる（既に一部あるが、**すべての非同期結果に適用**する）。

### 5.4 アプリ終了・未保存データ（ホストとコアの連携）

egui では `ViewportCommand::CancelClose` 等で終了を止められるが、分離後は **「終了要求 → コアが終了可否を返す → ホストが OS に応答」**の順が必要になる。連携漏れは**未保存の破棄**に直結するため、次を要件とする。

1. **終了シーケンス**: `can_close() -> Result<(), CloseBlockReason>` のように、コアが **ドメイン上のブロック理由**（例: 条件エディタ未保存）を返せること。
2. **ホストの責務**: OS／ウィンドウマネージャへの「終了キャンセル」はホストが実行。コアは **egui に依存しない**。
3. **テスト**: 未保存フラグが立っているときに `can_close` が失敗することをユニットテストで検証する。

### 5.5 再描画通知（dirty）と FPS

「状態が変わったので再描画」がアダプタとコアで分断されると、**通知漏れで画面が更新されない**／**過剰通知で CPU 占有**の両方が起きうる。対策として次を推奨する。

1. コアの更新戻り値に **`needs_repaint: bool`**（または `RepaintHint`）を含め、**論理的に変化したフレームだけ**真にする（または変化種別を列挙する）。
2. ポーリング待ち（分析ジョブ）は、現状どおり **`request_repaint_after`** 等で**間引き**し、完了時は明示的に再描画。
3. 無限ループ（毎フレーム状態が変わるバグ）を防ぐため、**同一ユーザー入力に対する二重適用**を防ぐガードをコアに置く余地を残す。

### 5.6 60 FPS との両立（指針）

- **1 フレーム内でやること**: 状態の反映、軽い計算、描画に必要なデータ準備。
- **1 フレーム内でやらないこと**: 大 CSV の全走査、大きな SQLite クエリの連発、ディスクの同期読み（可能ならバックグラウンドやバッチ化）。
- 既存の **セグメントキャッシュ**（`cached_segments` 等）のような **計算結果のメモ化**は、FPS 維持のため継続・整理してよい。

### 5.7 egui 即時モードと「1 フレーム遅れ」（アダプタ側の責務）

選択変更と `TreeScrollRequest` のような **スクロール指示**は、コアが「論理行インデックス」までを保証し、**ピクセル同期のスクロール・フォーカス**は egui アダプタが引き受ける。タイムラグで行が画面外に残るリスクは **中程度**（§11.2）であり、対策として (1) 可能なら **同一フレーム内**でスクロール適用、(2) それでも足りない場合は **次フレームでもう一度整列**する二段構え、(3) コアは **`selected_logical_id`（例: `unit_id`）**のような安定キーを持ち、インデックスだけに依存しない表示整合性を優先する、を検討する。

## 6. Tauri 移行のための API 契約（初期案・委任事項の具体化）

利用者は「任せる」としているため、以下を **初期の契約案**として置く。実装時に JSON の形に落とし込み、変更時は本書のバージョンを上げる。

### 6.1 コマンド（フロント → Rust / 例）

- `load_csv(path: string)` … 成否とメタ情報
- `set_filter(column, values)` … フィルタ状態更新
- `select_row(index)` … 選択変更
- `run_analysis(overrides)` … 分析ジョブ開始（ジョブ ID を返す）
- `open_db_viewer(paragraph_id)` … DB 参照用データ取得

### 6.2 イベント（Rust → フロント / 例）

- `analysis_progress { job_id, phase, message? }`
- `analysis_finished { job_id, outcome }`
- `error { code, message, job_id? }` … ジョブ起因のエラーは `job_id` を付与し、§5.3 の検証に使う。

**方針**: 非同期の完了・失敗・キャンセルに関わるイベントは **`job_id` 必須**とし、フロント／アダプタはコアの有効 ID と突き合わせる。

### 6.3 型の方針

- **serde** でシリアライズ可能な構造体に寄せる。
- コアの **公開モデル**（`AnalysisRecord` 等）をそのまま流用できるかは都度判断し、**UI 向けに細い DTO** を挟む余地を残す。

※ 現行 egui 版は当面この契約を **直接実装しない**。コアが「同じ情報を表現できる」状態になったときに、Tauri 側で実装する。

## 7. プラットフォーム・非機能

| 項目 | 方針 |
|------|------|
| OS | Windows のみ（パス区切り・ファイルダイアログ・検証環境を Windows に寄せる）。 |
| メモリ | 16GB 環境で実用できることを目安。厳密な上限値は設けない。 |
| 起動 | 特段の最適化は必須としない。 |
| 応答 | 操作に対するフィードバックを遅延させない。**重い処理はバックグラウンド**へ。 |
| FPS | **60 FPS** を目標に、メインスレッドの負荷を設計する。 |

## 8. テスト戦略（少しずつ）

1. **最優先**: フィルタ・レコード一覧の導出、CSV パース周辺など **純粋関数・データ変換**のユニットテスト。
2. **次点**: コアの **状態遷移**（イベント適用後の状態）をテスト（モック可能な副作用境界）。
3. **P2 以降で優先度を上げる項目**（セカンドオピニオンに基づく）:
   - **`all_records` と `filtered_indices` と `selected_row` の不変条件**（フィルタ再計算・レコード更新後に整合が取れているか）。
   - **メモ化キャッシュ**（セグメント等）の **無効化**: フィルタ変更、選択変更、該当レコード更新、ジョブ完了で期待どおり無効化されるか。
   - **古い `job_id` のイベントが無視される**こと。
   - **未保存時の `can_close`**（§5.4）。
4. **後回しでよい**: UI のスナップショットテスト、E2E（Tauri 化後に検討）。

## 9. 段階的移行計画（ロードマップ）

以下の順序を推奨する。各フェーズは「動作するアプリを維持したまま」小さく完了させる。

| フェーズ | 内容 | 成果物 |
|----------|------|--------|
| **P1** | `app.rs` の **論理分割**（ファイル・モジュール単位）。描画と状態更新のコメント境界の明確化。**インデックス整合・キャッシュ無効化の呼び出し箇所を一覧化**。 | 読みやすい構造、差分の影響範囲縮小。 |
| **P2** | **「コア状態」構造体**の抽出（egui 型を持たない）。更新は `Event` / `Command` 風に集約。**ジョブ ID 検証・キャッシュ無効化・終了可否**をコア責務としてカプセル化し、ユニットテストを追加。 | テスト可能な塊の出現。 |
| **P3** | **分析ジョブ・ファイルダイアログ**などを「ホスト側トレイト」または薄いラッパに集約。 | ジョブモデルと UI の疎結合。 |
| **P4** | **serde 可能な DTO** と IPC 草案の固定（本書の §6 を実装レベルに）。 | Tauri 移行時の着手材料。 |
| **P5** | Tauri プロジェクト新設・**最小画面**からの接続（Windows）。 | 実移行のパイロット。 |

長期目標の「本移行」は **P4 まで完了し、UI 再設計の方針が固まった段階**でゴール設定する。

以下は **実装時のチェックリスト**として使う。順序は依存関係に応じて入れ替えてよい。各タスクは **1 PR または 1 コミット単位**を目安とする。

#### P2: コア状態の抽出・イベント集約・テスト

| ID | タスク | 完了条件（例） |
|----|--------|----------------|
| P2-01 | **コア用モジュール**（仮名 `app_core` または `viewer_core`）を新設し、**egui / eframe 型を含まない** `struct` 定義から開始 | 空または最小状態でコンパイル |
| P2-02 | レコード・フィルタ・選択の **ドメイン状態**を `App` からコアへ移す（`egui::Context` を持たない） | ビルド成功、手動スモーク |
| P2-03 | **`Event` / `Command`（または `CoreMessage`）** 列挙型を定義し、ユーザー操作・ジョブ完了を **型で表現** | 主要パスが列挙子経由に |
| P2-04 | **`apply_event(&mut self, event) -> CoreOutput`** の形に寄せ、`CoreOutput` に **`needs_repaint`**（§5.5）を含める | 呼び出し側で `request_repaint` 連携 |

| P2-07 | **キャッシュ無効化**を `replace_records`・フィルタ変更・選択変更・注釈保存・ジョブ完了の**経路ごと**にコア内で明示 | ユニットテストで「別行のセグメントが残らない」ケース |
| P2-08 | **`filtered_indices` 再計算と `selected_row` のクランプ**を単一関数（または少数メソッド）に集約 | 不変条件のテスト追加 |
| P2-09 | **データソース世代**（§11.4）をコアに導入（最小: カウンタのみ）。DB Viewer・条件エディタが参照する世代をコアから供給 | 再読込後に不整合が検知できるか手動確認 |

## 10. リスクと対策

| リスク | 脅威度（目安） | 対策 |
|--------|----------------|------|
| 分離リファクタの過大化 | 中 | P1〜P2 で小さく切る。コアに入れない状態は「アダプタ専用」と明記する。 |
| 60 FPS 低下 | 中 | プロファイラでホットスポット特定。重処理の非同期化。§5.5 の再描画方針。 |
| Tauri API の手戻り | 低〜中 | 初期は「コアに載せる情報の完全性」優先し、フィールドは後から追加可能にする。 |
| **非同期ジョブと UI のレース**（古い結果で上書き） | **高** | §5.3 の `job_id`／世代による破棄。 |
| **終了時の未保存データ喪失** | **高**（発生時の影響大） | §5.4 の `can_close` とホスト連携。 |
| **キャッシュ無効化漏れ・インデックスズレ** | **高** | 不変条件をコアに集約、§8 のテスト。 |
| **スクロール／フォーカスずれ**（即時モード） | 中 | §5.7。安定キー（`unit_id` 等）の併用。 |
| **マルチウィンドウ・将来 IPC の状態不整合** | 中（長期で **高** に上がりうる） | §11.4、**セッション世代**・明示的リロード方針。 |

## 11. セカンドオピニオンに基づく脅威評価と対策要件（詳細）

外部レビューで指摘された 4 観点について、**脅威度**と**本設計での位置づけ**をまとめる。脅威度は「発生しやすさ × 影響の大きさ」の主観的な目安である。

### 11.1 総覧表

| ID | 観点 | 主な脅威 | 脅威度 | 設計上の対応（節） |
|----|------|----------|--------|-------------------|
| A | 非同期・ライフサイクル | 古いジョブ結果で状態を上書き（レース） | **高** | §5.3, §6.2, §8, §10 |
| B | 非同期・ライフサイクル | 終了時に未保存を破棄 | **高**（影響） | §5.4, §8, §10 |
| C | egui 分離・パフォーマンス | スクロール／選択の 1 フレーム遅れ、画面外に取り残す | **中** | §5.5, §5.7 |
| D | egui 分離・パフォーマンス | 再描画の漏れ／過剰でフリーズ感・FPS 低下 | **中** | §5.5, §7 |
| E | データ・計算 | メモ化キャッシュの無効化漏れ（別行の内容表示など） | **高** | §5.6, §8, P2 |
| F | データ・計算 | `filtered_indices` と `selected_row` の不整合 | **高** | §4.4, §8, P1〜P2 |
| G | マルチウィンドウ・将来分離 | DB Viewer／Condition Editor とメインのデータソースずれ | **中**（分散化で **高**） | 本節 11.4 |

### 11.2 UI 崩れ・パフォーマンス（観点 C, D）

即時モード UI では、コアが emit した論理状態と、egui がスクロールを適用するタイミングに **わずかなずれ**が生じうる。脅威度は **中**（ユーザー体験を損ねるが、データ破壊に直結しにくい）。**論理キー**（段落 ID 等）で表示内容の正しさを優先し、スクロールはアダプタで吸収する（§5.7）。

### 11.3 計算・データ整合（観点 E, F）

キャッシュとインデックスは **分離リファクタで最もバグりやすい**領域であり、脅威度 **高**。対策は「更新経路をコアの単一メソッドに集約し、そこで必ずキャッシュ無効化とフィルタ再計算を行う」**パターンに寄せる**こと。テストで **代表イベント列**を固定し、期待インデックス・期待テキストを検証する（§8）。

### 11.4 マルチウィンドウと状態同期（観点 G）

現状は単一 `App` 内で共有されているが、Tauri 化やプロセス分離後は **メインが読み込んだ CSV／パスと、サブ画面の認識**がずれやすい。脅威度は現状 **中**、**分散アーキテクチャ後は高**に上がる。

**方針（初期）**

1. **データソース世代（`session_id` または `load_generation`）**をコアが保持し、CSV 再読込時に単調増加させる。DB Viewer・条件エディタは **参照中の世代**を持ち、メインと不一致なら警告または自動クローズ／再読込を促す。
2. 条件 JSON を保存した場合、**分析ランタイムが参照するパス**とメインの認識を **明示的に同期**するイベント（「設定を再読込」相当）を設ける。
3. 長期は **「各ウィンドウはコアの投影」**であり、真実は常にコア 1 箇所、という不変条件を維持する。
