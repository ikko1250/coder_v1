# 「UI とアプリケーションコア分離・Tauri 移行ロードマップ 設計案」へのセカンドオピニオン

ご提示いただいた設計案（以下、本設計案）および現在のリポジトリのコード（特に `app.rs`、`analysis_runner.rs`、`model.rs` 等）を拝見しました。

現状の「UIとロジックの密結合（特に `app.rs` への肥大化）」という課題に対して、Tauri移行を見据えつつ、まずは段階的にRustコアを分離していく方針（P1〜P5のロードマップ）は非常に現実的であり、方向性として優れていると考えます。

その上で、現在の実装状況と本設計案を照らし合わせ、**バグの可能性、UI崩れ、計算ミスの可能性、意図しない動作の可能性** という4つの観点から、批判的かつ網羅的にフィードバックをまとめました。

---

## 1. バグの可能性 (非同期状態管理・ライフサイクル関連)

現在の実装では、非同期ジョブ（Pythonプロセスなど）とUIのやり取りが `mpsc` チャネルを通じて行われており、状態管理が `AnalysisRuntimeState` に組み込まれています。コア分離の際、以下の点でバグが発生しやすくなります。

* **ジョブ結果と最新のUI状態の不一致 (Race Condition)**
  * 現状の `app.rs` では `poll_analysis_job` が毎フレーム呼ばれ、結果を受け取ると即座に `App` の状態を上書きしています（`handle_analysis_success`等）。
  * コア（ドメイン状態）とUIアダプタを分離し、「コマンド・イベント」の形（設計案 §6）に移行する過程で、**「ユーザーがフィルタやファイル選択を変更した直後に、古いジョブの結果が届き、状態を意図せず上書きしてしまう」** リスクが高まります。
  * **回避策の例**: コア側で「現在の有効な Job ID」を厳密に管理し、イベント受信時に ID を検証して古いジョブ結果は破棄するロジックを組み込む。
* **終了時・クリーンアップのバグ**
  * `condition_editor_state.is_dirty` のような「未保存状態」がある場合、現状は `guard_root_close_with_dirty_editor` で egui の Viewport 終了をキャンセルしています。
  * これを「コア」と「UI・ホスト層」に分離すると、「ホストが終了コマンドを受け取る」→「コアに終了可能か問い合わせる」という非同期的なステップが必要になります。この連携が漏れると、**未保存のデータが破棄されてアプリが強制終了する** バグに繋がります。

## 2. UI崩れ・パフォーマンス低下の可能性 (egui 依存の分離)

本設計案では「短期的に egui を継続しつつコアを太らせる」としていますが、egui特有の即時モード（Immediate Mode）UIの仕様と分離する際に以下の問題が懸念されます。

* **スクロールやフォーカスの喪失**
  * 現状の `SelectionChange` や `TreeScrollRequest`、および `egui::ViewportCommand` (例: `focus_condition_editor_viewport`) は egui のライフサイクルに強く依存しています。
  * コア側から純粋な「選択状態の変更」を Emit しても、UI側（egui アダプタ）でそれを 1 フレーム遅れで検知してスクロール指示（`scroll_to_row`）に変換することになります。このタイムラグにより、**ツリーの選択行が画面外に取り残される、あるいは意図しない位置にスクロールが跳ぶ（UI崩れ）** リスクがあります。
* **再描画 (Repaint) ループの枯渇・暴走**
  * 現状、非同期処理の待ち受けでは `ctx.request_repaint_after(Duration::from_millis(100))` などを明示的に呼んでいます。
  * コアと UI アダプタが分離されると、「状態が変わったので再描画が必要」という通知（dirty flag）の管理が難しくなります。通知が漏れれば **画面がフリーズしたように見え**、過剰に通知すれば **FPSが低下し、CPU使用率が跳ね上がる** 可能性があります。

## 3. 計算ミス・データ不整合の可能性 (メモ化・フィルタリング)

機能パリティは求めないとしつつも、分析結果の表示などにおいて「計算ミスの可能性」に注意が必要です。

* **キャッシュ (メモ化) の無効化漏れによる古いデータの表示**
  * 現在、`app.rs` 内で `cached_segments` を用いて、選択行（`row_no`）に基づくパース結果をメモ化しています（`get_segments` メソッドなど）。
  * コア側に状態を移す際、フィルタ変更、行の選択変更、ジョブ完了などのイベントで適切にこのキャッシュを無効化（Invalidate）しないと、**違う行を選択しているのに前の行のテキスト（セグメント）が表示される** という計算・表示のミスが発生します。
* **インデックスのズレ (all_records と filtered_indices)**
  * 現状は `all_records` に対するインデックスの配列 `filtered_indices` を持ってフィルタリングを実現しています。
  * アプリケーションコアへこの状態を分離した際、もし「マニュアルアノテーションの追加（`save_annotation_for_selected_record`）」のようなイベントで `all_records` を更新し、同時に `filtered_indices` を再計算するステップが漏れた場合、**UIが指し示すインデックスと実際のデータがズレてしまい**、間違ったデータに対して操作を行ってしまう危険性があります。

## 4. 意図しない動作の可能性 (マルチウィンドウと状態同期)

設計案 §4.1 のレイヤ構造を導入する際、最も「意図しない動作」を生みやすいのが、現在複数あるウィンドウ/Viewport間の状態共有です。

* **マルチViewport (DB Viewer, Condition Editor) の状態非同期**
  * 現状のコードでは、`App` 構造体の中に `db_viewer_state` や `condition_editor_state` などがすべて同居しており、同じメモリ空間を毎フレーム直接参照しています。
  * これを Tauri のフロントへの移行（P5）や、それに向けた分離（P3, P4）を行うと、ウィンドウごとにプロセスやスレッドが分かれる（あるいはフロントエンドで別コンテキストになる）可能性があります。
  * 結果として、**メイン画面で別のCSVを読み込んだのに、開いたままの DB Viewer や Condition Editor が古いパスやデータを参照し続ける**、あるいは ** Condition Editor 側で JSON を保存したのに、メイン側の `analysis_runtime_state` がそれを検知できず古い設定で分析が走る** といった意図しない動作（状態の不整合）が起きやすくなります。

---

### 総評

設計案の「短期：保守性向上、長期：Tauri移行」という戦略は理にかなっています。
とくに **P1〜P2 (論理分割とコア状態の抽出)** のフェーズで、上述したような「キャッシュの無効化」「インデックスの整合性」「ジョブ結果の検証」をコア（ドメイン）側の責務としてカプセル化し、ユニットテストで守りを固めることが、安全な移行の鍵となります。回避策は最小限にとどめていますが、特に状態（State）を「更新する側」と「描画する側」に分けた際に生じるタイムラグと同期漏れに留意して実装を進めることをお勧めします。
